## H2 설계 옵션

### 옵션 A — 원래 등록 설계 그대로 진행

원래 H2 가설: "태스크 벡터의 합산/보간이 새로운 태스크로 전이된다"

H1a 실패에도 불구하고 H2 자체는 검증 가능하다. 공통 subspace가 없어도 벡터 합산은 가능하며 전이 효과가 있을 수 있다.

**장점**: 사전 등록된 경로를 그대로 따름  
**단점**: H1a 실패 → H2 성공의 연결 고리가 약해짐. "왜 공통 구조 없이 전이가 가능한가"를 새로 설명해야 함

### 옵션 B — H2를 직교성 검증으로 재설계

새로운 가설: "태스크 벡터의 완전 직교성이 task arithmetic의 간섭 없는 합산을 보장한다"

검증 방법:
1. 두 태스크 벡터 v_A + v_B를 모델에 적용
2. 태스크 A 성능 변화 측정 (v_B 추가가 A에 미치는 간섭)
3. 직교할수록 간섭이 작아야 함 → |cos(v_A, v_B)|와 성능 간섭의 상관관계

**장점**: H1a/H1b 결과와 직접 연결되는 실험  
**단점**: 사전 등록 경로 이탈, GPU 추가 필요

### 옵션 C — 논문 방향 전환, H2 생략

현재까지의 결과만으로 논문 기여를 구성:

"GRPO 기반 LoRA 태스크 벡터의 기하학적 분석:  
태스크 벡터는 도메인·포맷·기능 컴포넌트에 무관하게 완전 독립 방향을 가지며,  
이것이 task arithmetic의 수학적 안전성의 근거다."

**장점**: 추가 GPU 불필요, 깔끔한 스토리  
**단점**: H2 없이 전이 효과를 주장할 수 없음

---

## 현재 선호 방향

옵션 B를 수정한 형태:

1. 직교성이 실제 task arithmetic 성능에 어떤 영향을 미치는지 소규모 검증
2. 2~3개 태스크 쌍(ARC 쌍, 수학 쌍)에서 벡터 합산 후 cross-task evaluation
3. 결과를 H1a/H1b 결과와 연결해 "직교성 → 간섭 없음 → task arithmetic 안전" 주장 완성

예상 GPU 비용: 각 평가 런 30분 이내 (추론만, 추가 학습 없음)
